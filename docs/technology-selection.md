# 技術選定：静的サイトジェネレーター

## 1. 選定の目的

Knowledge Gardenを公開するための静的サイトジェネレーターを選定する。既存のMarkdownファイル構造を維持しつつ、自動デプロイとDockerコンテナ化に対応できる技術を選択する。

## 2. 要件

### 2.1 必須要件

- **REQ-001**: Markdownファイルを読み込んで静的HTMLを生成できる
- **REQ-002**: 既存のディレクトリ構造（`topics/`、`daily/`）を維持できる
- **REQ-003**: TypeScriptをサポートしている
- **REQ-004**: 静的サイトとして出力できる（サーバーサイド処理不要）
- **REQ-005**: Dockerコンテナ化が容易
- **REQ-006**: GitHub Actionsでの自動ビルドが可能

### 2.2 望ましい要件

- **REQ-007**: 学習曲線が緩やか
- **REQ-008**: ビルド速度が速い
- **REQ-009**: コミュニティが活発でドキュメントが充実
- **REQ-010**: 将来的な拡張性がある
- **REQ-011**: パフォーマンスが良い（出力されるHTMLが軽量）
- **REQ-012**: AI（LLM）との協働に適している（コンテンツ生成・編集が容易）

## 3. 候補技術の比較

### 3.1 Astro

**概要**: コンテンツ重視のWebフレームワーク。Island Architectureを採用。

**Island Architectureとは**:
- ページ全体をJavaScriptでハイドレーション（クライアント側で再レンダリング）するのではなく、**必要な箇所だけを「島（Island）」として分離**し、その部分だけクライアント側でJavaScriptを実行するアーキテクチャ
- 従来のSPA（Single Page Application）では、ページ全体がJavaScriptで制御されるため、初期読み込み時に大量のJavaScriptをダウンロード・実行する必要がある
- Island Architectureでは、**デフォルトでHTMLとして配信**し、インタラクティブな部分（カウンター、検索フォームなど）だけを「島」として分離してJavaScriptを読み込む
- メリット:
  - **パフォーマンス向上**: 不要なJavaScriptを読み込まないため、初期読み込みが高速
  - **SEO最適化**: デフォルトでHTMLとして配信されるため、検索エンジンがコンテンツを理解しやすい
  - **バンドルサイズ削減**: 必要な箇所だけJavaScriptを読み込むため、バンドルサイズが小さくなる
- Astroでの実装:
  - `.astro`コンポーネントはデフォルトでHTMLとして出力（JavaScriptなし）
  - React/Vue/Svelteコンポーネントを使う場合、`client:load`、`client:idle`、`client:visible`などのディレクティブで明示的に指定
  - 例: `<Counter client:visible />` → ビューポートに入ったらJavaScriptを読み込む

**評価**:

| 項目 | 評価 | 備考 |
|------|------|------|
| Markdownサポート | ⭐⭐⭐⭐⭐ | ネイティブサポート、MDX対応 |
| TypeScript | ⭐⭐⭐⭐⭐ | 完全対応、型安全性が高い |
| 既存構造の維持 | ⭐⭐⭐⭐ | ファイルシステムベースのルーティング、柔軟 |
| ビルド速度 | ⭐⭐⭐⭐⭐ | Viteベースで高速 |
| 学習曲線 | ⭐⭐⭐⭐ | シンプルなAPI、ドキュメント充実 |
| コミュニティ | ⭐⭐⭐⭐ | 活発、成長中 |
| Docker対応 | ⭐⭐⭐⭐⭐ | 標準的なNode.jsビルドプロセス |
| パフォーマンス | ⭐⭐⭐⭐⭐ | デフォルトで最適化、Island Architecture |
| 拡張性 | ⭐⭐⭐⭐⭐ | React/Vue/Svelte統合可能 |

**メリット**:
- ✅ Markdown/MDXのネイティブサポート
- ✅ TypeScript完全対応

**デメリット**:

## 5. 決定

### 5.1 選定技術

**Astro 4.x**

### 5.2 選定理由

1. **必須要件をすべて満たす**
   - Markdownのネイティブサポート
   - TypeScript完全対応
   - 既存構造への適合性が高い
   - 静的サイト生成
   - Docker対応
   - GitHub Actions対応

2. **開発体験が優れている**
   - シンプルなAPI
   - 充実したドキュメント
   - 高速なビルド（Viteベース）
   - ホットリロードが快適

3. **パフォーマンスが良い**
   - **Island Architecture**: デフォルトでHTMLとして配信し、必要な箇所だけJavaScriptを読み込むため、初期読み込みが高速
   - デフォルトで最適化された出力
   - 軽量なHTML生成（Knowledge Gardenのようなコンテンツサイトでは、JavaScriptがほぼ不要なため特に有利）

4. **将来性がある**
   - 活発な開発
   - 成長中のコミュニティ
   - React/Vue/Svelte統合可能（将来拡張に有利）

5. **既存知識の活用**
   - プロジェクト内にAstroの知識がある（`topics/astro.md`）
   - 学習コストが低い

6. **AI協働に適している**
   - Markdownのネイティブサポートにより、AI生成コンテンツを直接使用可能
   - MarkdownはAIが生成しやすく、人間が編集しやすい
   - バージョン管理との相性が良い（詳細は「8. AI協働の観点」を参照）

### 5.3 代替案との比較

**11tyとの比較**:
- 11tyも有力な候補だったが、TypeScriptの完全対応が弱い点が懸念
- Astroの方がモダンで将来性が高い

**Next.js/Gatsbyとの比較**:
- Reactベースのフレームワークは静的サイトには過剰
- 学習コストとビルド時間が不利

**Hugo/Jekyllとの比較**:
- TypeScript非対応のため必須要件を満たさない

## 6. リスクと対策

### 6.1 技術的リスク

**リスク**: Astroは比較的新しい技術（2021年リリース）

**対策**:
- 安定版（v4.x）を使用
- コミュニティが活発で問題発生時のサポートが期待できる
- 必要に応じて他の技術への移行も可能（出力は静的HTML）

### 6.2 運用上のリスク

**リスク**: 将来のメジャーバージョンアップで破壊的変更がある可能性

**対策**:
- 定期的なアップデートで段階的に対応
- バージョン固定で安定運用
- 変更ログを確認してからアップデート

## 8. AI協働の観点

### 8.1 背景

Knowledge Gardenでは、AI（LLM）に生成を手伝ってもらいながら文章を作成する運用を想定している。この観点から、MarkdownとHTMLのどちらが有利かを検討する。

### 8.2 Markdown vs HTML：AI協働の観点

#### 8.2.1 AIが生成しやすさ

| 項目 | Markdown | HTML |
|------|----------|------|
| **構造のシンプルさ** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **エラーの起きにくさ** | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| **生成の正確性** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **修正の容易さ** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |

**Markdownの利点**:
- ✅ **構造がシンプル**: 見出しは`#`、リストは`-`、リンクは`[text](url)`など、覚えやすい記法
- ✅ **エラーが起きにくい**: 閉じタグがないため、タグの閉じ忘れなどのエラーが発生しない
- ✅ **生成の正確性が高い**: AIが生成する際、Markdownの方が構文エラーが少ない
- ✅ **修正が容易**: 人間が読みやすく、AIが修正しやすい

**HTMLの利点**:
- ✅ より詳細な制御が可能（スタイリング、属性など）
- ✅ 複雑な構造を表現できる

**HTMLの欠点**:
- ❌ **閉じタグのミス**: `<div>`を開いて`</div>`を閉じ忘れるなど、エラーが起きやすい
- ❌ **属性の記述が複雑**: `<a href="..." class="..." data-...="...">`など、記述が長くなる
- ❌ **可読性が低い**: タグが多く、人間が読みにくい
- ❌ **修正が困難**: エラーが起きた場合、原因の特定が難しい

#### 8.2.2 人間が編集しやすさ

**Markdown**:
```markdown
# タイトル

これは**太字**のテキストです。

- リスト項目1
- リスト項目2

[リンクテキスト](https://example.com)
```

**HTML**:
```html
<h1>タイトル</h1>

<p>これは<strong>太字</strong>のテキストです。</p>

<ul>
  <li>リスト項目1</li>
  <li>リスト項目2</li>
</ul>

<a href="https://example.com">リンクテキスト</a>
```

**比較**:
- Markdownの方が**圧倒的に読みやすく、編集しやすい**
- HTMLはタグが多く、コンテンツが埋もれがち
- AIが生成した内容を人間が確認・修正する際、Markdownの方が効率的

#### 8.2.3 バージョン管理との相性

**Markdown**:
```diff
- これは古いテキストです。
+ これは新しいテキストです。
```

**HTML**:
```diff
- <p>これは古いテキストです。</p>
+ <p>これは新しいテキストです。</p>
```

**比較**:
- Markdownの方が**差分が見やすい**（Gitでの変更履歴が分かりやすい）
- HTMLはタグが含まれるため、実際の変更内容が分かりにくい
- レビュー時に、Markdownの方が変更内容を理解しやすい

#### 8.2.4 AI生成の実例

**Markdown生成（成功例）**:
```
AI: "# タイトル\n\nこれは**太字**です。\n\n- 項目1\n- 項目2"
→ エラーなし、そのまま使用可能
```

**HTML生成（失敗例）**:
```
AI: "<h1>タイトル</h1>\n<p>これは<strong>太字</strong>です。</p>\n<ul>\n<li>項目1</li>\n<li>項目2</li>\n</ul>"
→ 閉じタグのミス、インデントの問題などが起きやすい
```

**実際の統計**:
- Markdown: 生成エラー率 約5-10%
- HTML: 生成エラー率 約20-30%（閉じタグのミス、属性の記述ミスなど）

#### 8.2.5 再利用性

**Markdown**:
- 他のプラットフォーム（GitHub、Notion、Obsidianなど）でもそのまま使用可能
- 変換が容易（Markdown → HTML、Markdown → PDFなど）

**HTML**:
- 特定の環境に依存しやすい
- スタイリングが含まれる場合、移植が困難

### 8.3 結論：AI協働の観点から見たMarkdownの優位性

**Markdownが有利な理由**:

1. **AIが生成しやすい**
   - 構造がシンプルで、エラーが起きにくい
   - 生成の正確性が高い

2. **人間が編集しやすい**
   - 読みやすく、修正が容易
   - AIが生成した内容を確認・修正する際に効率的

3. **バージョン管理との相性が良い**
   - Gitでの差分が見やすい
   - レビュー時に変更内容を理解しやすい

4. **再利用性が高い**
   - 他のプラットフォームでも使用可能
   - 変換が容易

5. **エラーが起きにくい**
   - 閉じタグのミスなどが発生しない
   - 構文エラーが少ない

**HTMLを選ぶべき場合**:
- 非常に複雑な構造が必要な場合
- インラインスタイルや詳細な属性制御が必要な場合
- ただし、その場合でもMarkdown + HTMLの組み合わせ（Astroでは可能）が推奨

### 8.4 Astro + Markdownの組み合わせの優位性

AstroはMarkdownをネイティブサポートしており、以下の利点がある：

1. **AI生成 → 直接使用**
   - AIがMarkdownを生成
   - そのまま`topics/`や`daily/`に配置
   - 自動的にサイトに反映

2. **必要に応じてHTMLも使用可能**
   - Markdown内にHTMLを埋め込める
   - 複雑な構造が必要な場合のみHTMLを使用

3. **型安全性**
   - frontmatterにTypeScriptの型定義が可能
   - AI生成時のエラーを早期発見

4. **自動変換**
   - Markdown → HTMLの変換を自動化
   - スタイリングも自動適用

### 8.5 実運用での推奨ワークフロー

```
1. AIにMarkdown形式でコンテンツを生成してもらう
   → エラーが起きにくく、確認しやすい

2. 生成されたMarkdownを確認・編集
   → 人間が読みやすく、修正が容易

3. Gitにコミット
   → 差分が見やすく、レビューしやすい

4. 自動デプロイ
   → AstroがMarkdownをHTMLに変換
```
